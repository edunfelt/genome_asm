Investigation of data file

The file consists of 12 columns, of which only columns 1-2, 6-8 and 10-12 are of interest in this project. The first two columns represent the contig identifiers, columns 6 and 7 represent the start and end of the overlap in the first contig, and column 8 represent the length of the first contig. Columns 10 and 11 likewise represent the start and end of the overlap in the second contig, and the final column represents the length of this contig.

Because of the size of the data file, the best method to examine it is by using Unix tools, for example the Head command or the Less command. Upon inspection, it becomes clear that the contig identifiers are very long sequences of letters and numbers that, in this project, are largely meaningless. It would therefore be desirable to replace these with integers instead. We would also like to remove the columns that are not going to be used.

One way to pursue this could be to:

  • Trim the data file by:
    • identify and remove contigs completely contained in each other
    • replace identifiers with integers
    • remove unnecessary data
  • Read trimmed data into a graph data structure, for example by reusing code from lab6
  • Examine the resulting graph structure by finding the:
    • node degree distribution
    • number of connected components
    • size distribution of the connected components

Test files added

It will not be feasible to work directly with the original data, because of the large file size, and the risk of making mistakes. Therefore the first step is to write a couple of test files that follow the same structure as the original version of the data file. These files are stored in the subdirectory '01' under 'data' to indicate that they are our first batch of test files.

There are three test files, with a sketch attached that describes the overlaps and the resulting graph. In the first test case the result should be a connected graph and three contigs should be discarded. In the second case, the result is two connected components and three discarded contigs. The final case again describes one connected component and seven discarded contigs. There might be reason to adapt/extend these test cases when the code is ready.
